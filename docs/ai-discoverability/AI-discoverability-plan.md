# AI Discoverability Plan for m-dev-tools

**Document type:** Implementation plan
**Scope:** Make every artifact and tool in the `m-dev-tools` org discoverable
and usable by AI coding agents to the maximum extent
**Audience:** Org maintainers and the AI agents executing this plan
**Status:** Draft ŌĆö supersedes `AI-discoverability-guide.md` if accepted

---

## 1. Principles

1. **Repos own facts; the org owns routing.** Anything derivable from a repo's
   source must live in that repo as a generated artifact. The org-level catalog
   is allowed to contain *intent mappings* and *cross-references* only.
2. **The catalog is a build output, not a source.** A hand-edited catalog
   drifts. Treat `tools.json` like a compiled binary: generated from inputs,
   schema-checked, committed for cacheability, regenerated by CI.
3. **Optimize for the agent's actual workflow, not for completeness.** An agent
   needs (a) where to look, (b) how to execute, (c) how to verify. Recipes and
   verification gates earn their keep before exhaustive metadata.
4. **Drift is the enemy.** Every machine-readable claim is either trivially
   re-derivable or guarded by a CI check. No measurement without a
   `verified_on` date or `source_command`.
5. **Boring conventions over novel ones.** `llms.txt` per llmstxt.org.
   `AGENTS.md` with `CLAUDE.md` symlinked so Claude, Copilot, Codex, and
   Continue all find guidance without configuration. JSON Schema 2020-12 for
   contracts.

## 3. Layer 1 ŌĆö Per-Repo Contract

Every repo in the org must satisfy this contract before any org-level work
happens.

### 3.1 Mandatory artifacts per repo

| Artifact | Path | Purpose |
|---|---|---|
| Repo manifest | `dist/repo.meta.json` | Single file the org-level generator reads |
| Agent instructions | `AGENTS.md` (canonical) + `CLAUDE.md` symlink | Repo-local guidance for humans and AI |
| Drift gate | `make check-manifest` | CI fails if manifest is stale |

### 3.2 `repo.meta.json` schema (minimal contract)

```json
{
  "$schema": "https://m-dev-tools.github.io/schemas/repo.meta.v1.json",
  "id": "tool:m-cli",
  "repo": "https://github.com/m-dev-tools/m-cli",
  "role": "Canonical M CLI ŌĆö fmt / lint / test / coverage / lsp / doc",
  "language": ["python"],
  "license": "AGPL-3.0",
  "agent_instructions": "AGENTS.md",
  "verified_on": "2026-05-10",
  "exposes": {
    "commands":   "dist/commands.json",
    "lint_rules": "dist/lint-rules.json",
    "fmt_rules":  "dist/fmt-rules.json"
  },
  "consumes": ["tool:tree-sitter-m", "tool:m-standard"],
  "verification_commands": ["make check", "m doctor"]
}
```

`exposes` is the load-bearing field: a map from logical kind to a path or URL
where the actual structured data lives. The org catalog reads only
`repo.meta.json` and follows those pointers. It never has to know what a
"command" or "module" looks like internally.

### 3.3 Per-repo onboarding sequence

For each repo, in order:

1. Adopt `AGENTS.md` as canonical (move existing `CLAUDE.md` content into it;
   symlink `CLAUDE.md ŌåÆ AGENTS.md`).
2. Point at ŌĆö or add ŌĆö a `dist/` generator. Most repos already have one.
3. Write `dist/repo.meta.json` referencing existing artifacts. Do not invent
   new payloads; point at what's already generated.
4. Add `make check-manifest`: regenerate, diff, fail on diff.
5. Wire `check-manifest` into the existing CI workflow.

### 3.4 Current-state inventory and priority

| Repo | AGENTS.md | exposes payload | drift gate | priority |
|---|---|---|---|---|
| `m-stdlib` | needs check | `dist/stdlib-manifest.json` exists ŌĆö just wrap | partial | **tier 1** ŌĆö richest manifest |
| `m-standard` | needs check | `integrated/*.tsv` + `*.json` exist ŌĆö just wrap | needs add | **tier 1** ŌĆö language ground truth |
| `m-cli` | exists | needs `m capabilities --json` and `m help --json` | needs add | **tier 1** ŌĆö single biggest leverage win |

## 4. Layer 2 ŌĆö Org Routing

Built only after tier-1 repos satisfy the ┬¦3 contract.

### 4.1 What the meta-repo holds

| File | Role | Hand-edited? |
|---|---|---|
| `profile/llms.txt` | AI doorway, llmstxt.org-compliant | yes |
| `profile/README.md` | Human overview | yes |
| `profile/tools.json` | **Generated** org catalog | **no** |
| `profile/tools.schema.json` | Schema for `tools.json` | yes (rarely) |
| `profile/task_index.json` | Hand-curated intentŌåÆtyped-ID map | **yes** (only hand-curated routing) |
| `profile/repo.meta.schema.json` | Schema for per-repo manifests | yes (rarely) |
| `profile/build/build-catalog.py` | Generator | yes |
| `profile/build/validate-catalog.py` | jsonschema validator | yes |
| `profile/build/test-discovery-protocol.py` | E2E handshake test | yes |

The split between **generated `tools.json`** and **hand-curated
`task_index.json`** is the most important design choice. Intent mapping is
irreducibly human work; everything else is mechanical.

### 4.2 Generation flow

```
For each repo in the org:
  fetch repo.meta.json
  validate against repo.meta.schema.json
  fetch each exposes.* artifact, validate against its kind schema
  copy a SUMMARY view into tools.json (counts, categories, pointer URL)
  do NOT inline the full payload

Merge with task_index.json (hand-curated)
Validate the merged tools.json against tools.schema.json
Emit deterministically (sorted keys, stable spacing)
Commit if changed
```

### 4.3 What `tools.json` keeps, what it sheds

**Keeps** (routing-only):

- `org` block,
- one `tools[id]` entry per repo with `repo`, `role`, `license`,
  `agent_instructions`, `verified_on`, summary counts, **pointer URLs only**,
- `task_index` (merged in from `task_index.json`),
- `workflow` (procedural recipes: scaffold, TDD loop, deploy),
- `discovery_protocol` (the handshake spec).

**Sheds** (drifting detail):

- inlined `subcommands` block ŌåÆ replaced by `commands_url`,
- inlined `modules` block ŌåÆ replaced by `manifest_url` + `module_count` +
  `categories`,
- inlined lint rule lists ŌåÆ replaced by `lint_rules_url`,
- inlined parser stats ŌåÆ replaced by `parser_metadata_url`.

Net effect: `tools.json` shrinks from ~350 lines to ~120 lines and stops being
the source of drift.

### 4.4 Typed IDs

Single enforced grammar:

```
^(tool|cmd|module|rule|doc|data|workflow|task|recipe):[a-z0-9_-]+(#[A-Za-z0-9._-]+)?$
```

Examples: `tool:m-cli`, `cmd:m-cli#test`, `module:m-stdlib#STDJSON`,
`rule:m-cli#M-MOD-009`, `workflow:tdd_inner_loop`, `recipe:new-app-tdd-ci`.

The schema must enforce this; no string in `tools.json` that smells like a
reference may bypass the regex.

### 4.5 Schema discipline

- `additionalProperties: false` at every level with an explicit allow-list for
  additive fields; promote to `unevaluatedProperties: false` once the contract
  is stable.
- Split version into two fields: `schema_compat: 1` (integer, only bumps on
  breaking changes) and `schema_version: "1.3"` (display).
- `tool` entries require: `id`, `repo`, `role`, `license`,
  `agent_instructions`, `verified_on`.
- `task_index` leaves require: `intent`, `primary` (typed ID).
- `workflow.steps[]` require: `n`, `action`, and either `command` or `tool`.

### 4.6 `llms.txt` ŌĆö strict llmstxt.org form

```markdown
# m-dev-tools

> Engine-neutral M (MUMPS) developer tools ŌĆö TDD/CI/lint/fmt/test/coverage
> across YottaDB and InterSystems IRIS. Modeled on Go/Rust/Python toolchains.

## Start here

- [Org catalog (machine-readable)](.../tools.json): intent ŌåÆ repo routing
- [Schema](.../tools.schema.json): contract for tools.json
- [Org README](.../README.md): human overview

## Core repositories

- [m-cli](.../m-cli/AGENTS.md): canonical `m` CLI surface
- [m-stdlib](.../m-stdlib/AGENTS.md): runtime standard library
- [m-standard](.../m-standard/AGENTS.md): language reference data
- [tree-sitter-m](.../tree-sitter-m/AGENTS.md): parser

## Guardrails

- Do not invent STD* APIs or `m` commands; route the gap to the owning repo.
- Use `m-stdlib` before adding helper logic elsewhere.
- For language semantics, query `m-standard`'s TSVs/JSON, not memory.
```

Forty lines max. Anything longer lives in README or `tools.json`.

## 5. Layer 3 ŌĆö Agent Operability

### 5.1 Recipes (executable, not stubs)

Each recipe is one Markdown file in `.github/docs/recipes/` with frontmatter
declaring the typed IDs it touches, and a body that lists *exact* commands and
expected outputs. A recipe is "done" only when running its commands top-to-
bottom on a fresh clone produces the documented result.

Initial seven, in priority order:

1. **`new-app-tdd-ci.md`** ŌĆö `m new fetcher && cd fetcher && m test && m fmt
   --check && m lint --error-on=error && m coverage --min-percent=85`. Verifies
   the entire toolchain end-to-end.
2. **`use-stdlib-module.md`** ŌĆö How an agent looks up `parse^STDJSON`, finds
   an example, and calls it.
3. **`add-stdlib-module.md`** ŌĆö Repo-side recipe: tests first, implement,
   regenerate manifest, verify drift gate.
4. **`add-lint-rule.md`** ŌĆö Repo-side recipe in `m-cli`.
5. **`add-m-cli-command.md`** ŌĆö Repo-side recipe in `m-cli`.
6. **`investigate-failure.md`** ŌĆö `m test` failed; the standard triage path.
7. **`add-editor-support.md`** ŌĆö VS Code extension changes.

Each recipe that *can* be CI'd is. Recipes that can't (e.g. publishing a VS
Code extension) declare so and link to a manual checklist.

### 5.2 Discovery protocol handshake test

One script that mechanically exercises the ┬¦13 success criteria of the prior
guide and the ┬¦7 criteria below:

```python
# profile/build/test-discovery-protocol.py
1. Fetch llms.txt ŌåÆ parses, all links resolve.
2. Fetch tools.json ŌåÆ validates against tools.schema.json.
3. Match a sample intent against task_index ("parse JSON in M").
4. Resolve the primary typed ID through tools.json.
5. Fetch the pointed-at manifest (e.g. stdlib-manifest.json).
6. Find STDJSON in the manifest ŌåÆ assert it has signature + example.
7. Print the routing trail; fail on any broken link or schema violation.
```

This is the only check that proves the system works end-to-end. It runs in CI
on every push and on a weekly schedule to catch external link rot.

### 5.3 MCP server

A thin MCP server wrapping `tools.json` + `task_index.json` + the discovery
protocol. Three tools:

- `route_intent(query: str) ŌåÆ typed_id[]`
- `describe(typed_id: str) ŌåÆ JSON blob with pointers`
- `verify(repo: str) ŌåÆ runs that repo's verification_commands; returns status`

One Python file (~150 LOC). Turns the catalog into a first-class protocol
surface instead of "go read these files."

**Distribution.** Originally GitHub-Release-wheel-only (Phase 4 v0.1.0
shipped that way); the **PyPI deferral was reversed at Phase 6
planning** because publishing to the official MCP registry
(`registry.modelcontextprotocol.io`) requires the server be reachable
through a registry-supported channel ŌĆö and PyPI is the natural fit for
a Python package. The original concern (permanent PyPI namespace +
yank-only versions) is now mitigated by Phase 4 having already
shipped a real-agent-tested v0.1.0 via the GitHub Release path; the
API surface (`route_intent` / `describe` / `verify`) is no longer
speculative.

Channels, in order of recommended use:

1. **PyPI** ŌĆö `pip install m-dev-tools-mcp` or
   `uvx m-dev-tools-mcp`. This is the primary channel and what the
   MCP registry's `server.json` declares.
2. **GitHub Release wheel** ŌĆö `pip install
   https://github.com/m-dev-tools/m-dev-tools-mcp/releases/download/v<X.Y>/m_dev_tools_mcp-<X.Y>-py3-none-any.whl`.
   Stays available for environments that can't reach PyPI.
3. **Source via `uvx` + git tag** ŌĆö
   `uvx --from git+https://github.com/m-dev-tools/m-dev-tools-mcp@v<X.Y> m-dev-tools-mcp`.
   Still works; useful for testing pre-release commits.

The MCP-client config (Claude Desktop / Codex / Continue / etc.)
points at the `m-dev-tools-mcp` binary any install path provides.
See Phase 6 below for the registry listing that makes config-less
discovery possible.

## 6. Continuous Enforcement

The system is worthless if any layer is allowed to drift. Enforcement runs at
three altitudes.

### 6.1 Per-repo CI (Layer 1)

- `make check-manifest`: regenerate `dist/repo.meta.json` and `exposes.*`,
  diff against committed copy, fail on diff.
- `make check-agents`: assert `AGENTS.md` exists and contains required
  sections (setup, test, verify).

### 6.2 Meta-repo CI (Layer 2)

- `make catalog`: regenerate `tools.json` from current repo manifests; fail
  on diff.
- `make validate-catalog`: real jsonschema validation
  (`check-jsonschema` or `python-jsonschema`), not `python -m json.tool`.
- `make test-protocol`: run the discovery handshake test.
- `make check-freshness`: fail if any `verified_on` is older than 90 days.
- `make check-links`: HEAD every URL in `tools.json` and `llms.txt`; fail on
  404.
- `make check-licenses`: `gh api repos/m-dev-tools/<repo>` and reconcile the
  `license` field.

### 6.3 Schema versioning enforcement

A dedicated CI check: if `tools.schema.json` or `repo.meta.schema.json`
changes and the diff isn't purely additive, require `schema_compat` to bump
and require a row in a `schema-changelog.md`.

## 7. Phased Execution

Each phase has a single exit criterion. Don't start phase N+1 until N is green.

### Phase 0 ŌĆö Tier-1 per-repo contract (Layer 1)

Goal: `m-cli`, `m-stdlib`, `m-standard` each emit a validated
`dist/repo.meta.json`, have `AGENTS.md`, and have `make check-manifest`.

Parallel work across the three:

- `m-stdlib`: wrap existing `stdlib-manifest.json` with a top-level
  `repo.meta.json`. Add `make check-manifest`. ~1 day.
- `m-standard`: emit `repo.meta.json` pointing at existing `integrated/*`
  tables. ~1 day.
- `m-cli`: implement `m capabilities --json` and `m help --json` (emit
  `dist/commands.json` and `dist/lint-rules.json`). Drop `repo.meta.json`
  wrapper. ~3ŌĆō5 days.

**Exit:** a small driver in `.github/profile/build/` fetches all three
`repo.meta.json` files, validates them, and follows their `exposes` pointers
without error.

### Phase 1 ŌĆö Org routing layer (Layer 2)

Goal: `tools.json` is generated, schema-strict; `task_index.json` is the only
hand-curated routing source.

- Define and publish `repo.meta.schema.json`.
- Tighten `tools.schema.json`: typed-ID regex, `additionalProperties: false`,
  `task_index` inner schema.
- Write `build-catalog.py`, `validate-catalog.py`.
- Split out `task_index.json` from `tools.json`; merge at build time.
- Rewrite `llms.txt` to llmstxt.org format.
- Wire `make catalog`, `make validate-catalog`, and a CI workflow.

**Exit:** `make catalog && make validate-catalog` is green; `tools.json` no
longer contains hand-maintained subcommand/module/rule lists.

### Phase 2 ŌĆö Tier-2 repos (Layer 1 cont.)

Goal: `tree-sitter-m`, `m-test-engine`, `m-modern-corpus` ship
`repo.meta.json`.

Smaller scope per repo. **Exit:** all three appear in generated `tools.json`
with `verified_on` dates within 30 days.

### Phase 3 ŌĆö Recipes + handshake test (Layer 3)

Goal: at least 4 of the 7 priority recipes are written and CI-verified; the
discovery handshake test runs in CI.

This is the highest-leverage *agent-facing* deliverable and should not be
deferred to last.

**Exit:** an agent (or `test-discovery-protocol.py`) can complete the
new-app-TDD-CI recipe top-to-bottom against a fresh clone, with green CI.

### Phase 4 ŌĆö Tier-3 repos + MCP server

Goal: VS Code extensions and `m-cli-extras` ship `repo.meta.json`. Ship
`m-dev-tools-mcp` as a GitHub-Release wheel ŌĆö PyPI follows in Phase 6
once registry-listing requires it (Phase 4 closed pre-PyPI per the
original ┬¦5.3 deferral; ┬¦5.3 was updated at Phase 6 planning to
reverse the deferral).

**Exit:** Claude/Codex/Continue can be pointed at the MCP server (installed
via `pip install https://github.com/m-dev-tools/m-dev-tools-mcp/releases/download/v<X.Y>/...whl`
or `uvx --from git+...@v<X.Y>`) and answer `route_intent("parse JSON in M")`
with `module:m-stdlib#STDJSON`.

### Phase 5 ŌĆö Continuous enforcement hardening

Goal: freshness, link-check, license-reconcile gates running weekly in CI.

**Exit:** a stale catalog or broken link is caught within 7 days.

### Phase 6 ŌĆö Distribution surface (broaden client reach)

Goal: ship `m-dev-tools-mcp` through every distribution channel an
MCP client is likely to consult, so a user doesn't have to
hand-roll a `.mcp.json` to use it.

Required ships:

1. **PyPI publication.** `m-dev-tools-mcp` ships as a regular Python
   package via `uv publish`. Reverses the ┬¦5.3 deferral ŌĆö see that
   section's updated rationale. PyPI is the official MCP registry's
   accepted Python package channel; the registry-listing step
   below references the PyPI entry.
2. **Official MCP registry listing.** Publish
   `io.github.m-dev-tools/m-dev-tools-mcp` to
   [`registry.modelcontextprotocol.io`](https://registry.modelcontextprotocol.io/)
   via the `mcp-publisher` CLI. Adds a `server.json` to the
   `m-dev-tools-mcp` repo root with `registryType: pypi`;
   `mcp-publisher publish` lands it in the registry. Once listed,
   any registry-aware client (Codex, Continue, Goose, ŌĆ”)
   auto-discovers without `.mcp.json` config.
3. **Maintenance:** every release tag (`v0.1.1`, `v0.2.0`, ŌĆ”)
   triggers `uv publish` to PyPI **and** `mcp-publisher publish` to
   the registry from a GitHub-Actions workflow in `m-dev-tools-mcp`.
   PyPI auth via Trusted Publisher (OIDC, no stored tokens);
   `mcp-publisher` auth via GitHub OIDC mode for the same reason.

Optional (deferred until external adoption demand):

- **VS Code extension** bundling the MCP server for the
  VS Code / Cursor audience specifically. Sibling to the existing
  `tree-sitter-m-vscode` and `m-stdlib-vscode` Tier-3 repos.

**Exit:** a fresh-install MCP-capable client points at the registry
and resolves `route_intent("parse JSON in M")` without any
hand-written client-side config. The catalog's `tools.json` entry
for `m-dev-tools-mcp` carries a `registry_url` field pointing at
the live registry record.

## 8. Explicit Non-Goals

1. **No project-management tracker** (no implementation-tracker, no parallel
   ADR log, no remediation backlog) in the meta-repo. Use GitHub Projects
   and Issues. A parallel tracking universe is overhead without leverage.
2. **No `AI-INDEX.md`** as a separate doc. `tools.json` is the index;
   `AI-INDEX.md` is just a worse rendering of the same data.
3. **No org-level CONTRIBUTING.md duplication of repo content.** The org
   `CONTRIBUTING.md` is ~30 lines pointing at each repo's own contribution
   guide.
4. **No history/archive routing in the catalog.** `m-tools` is archived;
   its three design docs are rehosted under
   [`.github/docs/history/`](history/) and routed via `task_index.json`'s
   `history` category as `doc:m-dev-tools#<slug>` typed IDs. The
