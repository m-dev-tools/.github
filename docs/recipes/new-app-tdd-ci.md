# Scaffold a new M project with TDD + CI in 60 seconds

## Prerequisites

- `m` (m-cli) on `$PATH`. Check with `m --version`; if missing, install
  per the [m-cli README](https://github.com/m-dev-tools/m-cli#install).
- `m doctor` exits 0 — confirms YottaDB, tree-sitter-m, m-standard TSVs,
  and the m-stdlib manifest all resolve.
- An engine reachable by `m test`: either a local `mumps` binary, or
  the `m-test-engine` Docker container (auto-detected — see
  [`tool:m-test-engine`](https://github.com/m-dev-tools/m-test-engine)).
- Network access for `m new` and the engine bootstrap on first run.

The recipe runs in a fresh `mktemp -d`. It writes no state outside the
temp dir. `m new` scaffolds a `routines/` + `tests/` layout plus a
generated Makefile; the recipe drives the tooling through that
Makefile's `fmt-check`, `lint`, `test`, and `coverage` targets so the
exact path arguments stay in sync with the scaffold.

## Steps

```bash
# step 1 — scaffold a fresh project named "fetcher"
m new fetcher

# step 2 — enter the project root before any per-project command
cd fetcher

# step 3 — assert the scaffold is already canonically formatted
make fmt-check

# step 4 — run the default lint profile via the scaffold's wrapper.
#   The Makefile invokes `m lint --error-on=error routines tests`, so
#   info/warning findings are allowed; errors fail the gate.
make lint

# step 5 — run the unit-test suite generated by `m new`. Green on
#   first run because the scaffold ships a passing FETCHERTST.m.
make test

# step 6 — enforce the 85% coverage gate the global CLAUDE.md
#   mandates. The Makefile target wires --routines / --tests for us.
m coverage --routines routines --tests tests --min-percent=85
```

## Expected output

The output of `m new fetcher` must contain: `Scaffolded fetcher`.
The output of `make fmt-check` must contain: `unchanged`.
The output of `make lint` must contain: `0E`.
The output of `make test` must contain: `1/1 assertions passed`.
The output of `m coverage --routines routines --tests tests --min-percent=85` must contain: `100.0%`.

## Failure triage

If any of the commands above exits non-zero, see
`recipe:investigate-failure` (post-exit Phase 3 recipe — tracked under
docs/recipes/README.md's seven-recipe table). Briefly:

- **`m new` fails** — usually a stale m-cli install. Re-run `m doctor`;
  upgrade with `uv pip install -U m-cli` from the m-cli checkout.
- **`make test` fails** — confirm an engine is reachable with `m doctor`.
  Docker-backed runs need `m-test-engine` up: `make -C
  ~/m-dev-tools/m-test-engine up`.
- **`make fmt-check` fails** — re-run `make fmt` to write the
  canonicalisation, then diff the result.
- **`make lint` fails** — `m lint --rules=default --format=json
  routines tests | jq` shows per-rule findings. Fresh scaffolds should
  produce zero error-severity findings; any fired is a regression in
  m-cli.
- **`m coverage --min-percent=85` fails** — the scaffold ships a single
  test covering the single label, so dipping below 85% means `m
  coverage` lost track of either the routine or the suite (check
  `m doctor`'s "routines path" line).
